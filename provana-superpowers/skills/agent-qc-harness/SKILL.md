# Agent QC Harness

Cross-cutting skill for PM/QA verification of AI agent outputs across all three pods. This is not a manual testing skill — PM/QA designs the verification system and the BMAD QA agent runs it. Human judgment provides the final pass/fail decision.

**Announce at start:** "Running agent-qc-harness to verify agent output against story ACs."

## PM/QA philosophy at Provana

The person who wrote the spec is the right person to verify whether the output matches it. AI-generated code has a measurably higher logic error rate than human-reviewed code. Human judgment on outcomes is non-optional — not a concession to traditional QA, but a fundamental requirement of AI-native delivery.

## Step 1: Load story ACs

Read the story file (`docs/story-NNN.md`) and extract all acceptance criteria. These are the test cases.

For each AC:
- Is it specific and testable? If not, surface this to PM/QA before running verification.
- Does it have a measurable pass/fail condition? If not, work with PM/QA to add one.

## Step 2: Generate test cases from ACs

BMAD QA agent generates test cases from each AC:

```python
# Generated by BMAD QA agent from story-NNN.md ACs
test_cases = [
    {
        "ac_ref": "AC1",
        "description": "[AC text]",
        "input": "[test input]",
        "expected_output": "[exactly what should happen]",
        "pass_condition": "[measurable criterion]",
        "fail_condition": "[what constitutes failure]",
    },
    ...
]
```

## Step 3: Run agent output verification

For each test case, run the agent and evaluate:

```python
# tests/qc/test_story_NNN_acs.py
class TestStoryNNN:
    """Auto-generated from story-NNN.md ACs by BMAD QA agent."""

    def test_ac1_[description](self):
        result = agent.run(test_cases[0]["input"])
        assert meets_condition(result, test_cases[0]["pass_condition"])
        assert not meets_condition(result, test_cases[0]["fail_condition"])
```

## Step 4: Hallucination detection

For all pods, run the hallucination detector after agent output is produced:

```python
# src/qc/hallucination_detector.py
class HallucinationDetector:
    """
    Compares AI output against ground truth.
    Producer: Core team. Consumer: PM/QA and Tech FDE.
    """
    def check(self, ai_output: str, ground_truth: GroundTruth) -> HallucinationReport:
        return HallucinationReport(
            fabricated_facts=self._find_unsupported_claims(ai_output, ground_truth),
            confidence_violations=self._find_overconfident_claims(ai_output),
            scope_violations=self._find_out_of_scope_claims(ai_output, ground_truth),
            verdict=HallucinationVerdict.PASS if not any_violations else HallucinationVerdict.FAIL
        )
```

**Zero tolerance for hallucinations in:**
- Conv.AI: any factual claim about the customer's account
- Doc.AI: any extracted field value
- BPM: any process instruction or compliance statement

## Step 5: Domain-specific QC tools

### Pod 1 — Voice harness QC

```bash
# Run voice quality rubric against latest build
python tools/voice_quality_rubric.py \
  --pipeline src/conv_ai/pipelines/[name]_pipeline.py \
  --test-set tests/conv_ai/acceptance/ \
  --output reports/voice-qc-[date].md
```

Checks:
- Intent accuracy on acceptance test set (pass: >90%)
- Escalation accuracy (pass: 100% on explicit requests)
- Turn latency (pass: <500ms p95)
- TTS naturalness (pass: MOS >3.5)

### Pod 2 — Extraction QC

```bash
# Run extraction accuracy against ground-truth dataset
python tools/extraction_qc.py \
  --pipeline src/doc_ai/pipelines/[name]_pipeline.py \
  --ground-truth tests/doc_ai/ground_truth/ \
  --output reports/extraction-qc-[date].md
```

Checks:
- Per-field accuracy (pass: >85% each field)
- Missed fields rate (pass: <5%)
- Below-confidence fields (pass: flagged, not silently passed)

### Pod 3 — Process compliance QC

```bash
# Run SOP compliance test suite
python tools/bpm_compliance_qc.py \
  --agent src/bpm/agents/sop_[name]_agent.py \
  --sop docs/[name]-sop.yaml \
  --test-cases tests/bpm/acceptance/ \
  --output reports/bpm-qc-[date].md
```

Checks:
- Deterministic steps: 100% SOP compliance
- Judgment steps: all flagged to human (none silently resolved)
- Exception handling: correct escalation path followed

## Step 6: Human judgment — pass/fail decision

Present QC report to PM/QA:

```markdown
## QC Report — Story NNN: [Story Name]
**Date:** YYYY-MM-DD
**Agent build:** [git commit hash]

### AC verification results
| AC | Test result | Notes |
|----|-------------|-------|
| AC1 | ✅ PASS | [notes] |
| AC2 | ❌ FAIL | [specific failure description] |

### Hallucination check
**Result:** [PASS / FAIL]
[Details if FAIL]

### Domain QC
[Pod-specific results]

### PM/QA verdict
[ ] PASS — all ACs met, no hallucinations, domain QC passed
[ ] CONDITIONAL PASS — [specify conditions]
[ ] FAIL — [specify what needs fixing]

**PM/QA signature:** [name] [date]
```

The PM/QA human partner makes the final call. AI output is not self-certifying.

## Step 7: Azure DevOps CI gate

The `post-test.sh` hook blocks merge if:
- Any AC test fails
- Hallucination detector reports a violation
- Domain QC pass threshold not met
- PM/QA verdict is not PASS or CONDITIONAL PASS

This is the Azure DevOps CI gate on PR referenced in the Provana delivery model.
